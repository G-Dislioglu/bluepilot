TITLE: Builder migration wave 2 - database foundation
DATE: 2026-05-31

## Summary

BP-128 adds the Bluepilot Builder database foundation.

It migrates the complete Builder schema into `builder/src/schema/builder.ts`, adds a
builder-local Neon/Drizzle `getDb()` using `BLUEPILOT_BUILDER_DATABASE_URL`, and migrates only
the two DB-only modules in scope: `poolState` and `builderApprovalArtifacts`.

No live database is required for local tests.

## Claude Package Corrections

The supplied BP-128 package was corrected before implementation:

- The contract needed WLP-valid `reuse_target` and `evidence_required` fields.
- `STATE.md` and `docs/SESSION-LOG.md` were added to the allowed scope because Bluepilot's
  ANKER-REGEL requires anchor sync after a block.
- `poolState.ts` and `builderApprovalArtifacts.ts` could not be byte-identical in their import
  lines because they moved from `server/src/lib` to `builder/src`. Only the relative imports to
  `db.ts` and `schema/builder.ts` were relocated.
- `opusErrorLearning` stayed out because it imports `providers`.

## Infra Step For Later

When the Builder is ready for live DB-backed operation, create a dedicated Bluepilot Builder
database, set:

```bash
BLUEPILOT_BUILDER_DATABASE_URL=postgresql://user:password@host/database?sslmode=require
```

Then add or run the appropriate Drizzle schema push/migration command from `builder/`. This task
does not create the database and does not push the schema.
