# BP-146 Review Packet - Permit-gated Sandbox Write Runbook

## Summary

BP-146 adds a narrow live 3C trigger for one permit-gated sandbox write.
It corrects the Claude proposal by avoiding the old orchestrator-based real-write route:
the new endpoint calls `smartPush` directly with fixed sandbox coordinates and a caller-provided
`permitId`.

No live write was performed in this task.

## Critical Corrections

| Claimed/initial idea | Correction |
|---|---|
| Use the existing `/probe/sandbox-real-write` path for 3C | Rejected: that path still uses `orchestrateTask` and does not pass `writePermit` to `smartPush`. |
| Let the request describe the target | Rejected: repo, branch, path, op, and baseSha are hard-coded. |
| Permit content can be typed into multiple commands | Rejected in runbook: one UTF-8 file is the single source of bytes. |

## Safety Invariants

- Route is closed unless `BLUEPILOT_SANDBOX_PERMIT_WRITE_ENABLED=true`.
- Confirm phrase must be exactly `permit-write-to-bluepilot-sandbox`.
- Request body may contain only `confirm`, `permitId`, and `contentBase64`.
- Target is fixed to `G-Dislioglu/bluepilot-sandbox`, branch `main`,
  path `.bluepilot/phase-3c-permit-write.md`, op `create`, baseSha empty.
- Response reports length and status, but never serializes content, base64 content, or token values.
- The existing BP-145 `smartPush` byte-output guard remains responsible for computing the final content hash
  and consuming the permit through the Maya corridor before GitHub write.
