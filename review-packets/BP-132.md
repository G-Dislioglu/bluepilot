# BP-132 Review Packet - Builder Runtime Health Entry

## Scope

BP-132 adds the minimal runtime process that a later Render service can start. It exposes health
checks only; it does not expose any Builder execution route.

## Files Changed

| File | Reason |
|---|---|
| `contracts/BP-132.json` | Task contract; `baseline_ref` corrected to `5716470` |
| `builder/src/health.ts` | Testable health and DB-readiness logic |
| `builder/src/server.ts` | Minimal `node:http` server reading `PORT` |
| `builder/tests/health.test.ts` | Liveness, missing-DB, and failing-DB behavior |

`builder/package-lock.json` is unchanged.

## Security Boundary

`builder/src/server.ts` imports only:

```text
node:http
./health.js
```

`builder/src/health.ts` imports:

```text
node:http types
drizzle-orm sql
./db.js
```

No HTTP route imports `opusTaskOrchestrator`, `opusBuildPipeline`, `builderExecutor`,
`builderGithubBridge`, or `mayaBuilderGateClient`. BP-132 cannot trigger a build over HTTP.

## Routes

| Route | Expected behavior |
|---|---|
| `GET /health` | Returns HTTP 200 with liveness JSON. Does not call `getDb()`. |
| `GET /health/db` | Runs a small DB-readiness probe. Returns HTTP 200 `reachable`, or HTTP 503 `not_configured`/`unreachable`. |

```text

```

Real Neon DB reachability was verified through the Neon MCP without writing the connection string
to a shell command:

```text
SELECT 1 AS ok -> ok = 1
```

The combined Render-style `/health/db -> reachable` check is intentionally left for the later
Render service setup, after `BLUEPILOT_BUILDER_DATABASE_URL` is set as a secret in Render.

## Later Render Settings

```text
Root Directory: builder
Build Command: npm install && npm run typecheck && npm test
Start Command: npm start
```

Required secret:

```text
BLUEPILOT_BUILDER_DATABASE_URL=<Neon connection string for bluepilot-builder/neondb>
```

## Result

BP-132 gives Render a real Bluepilot Builder process to start while preserving the hard boundary:
health only, no Builder execution route.
